## Functional

### Introduction

This repo (`statox/dotfiles`) is a personal Linux dotfiles and workstation-setup
repository. It manages:

- Symlinked config files for the shell, desktop environment, editor, and terminal.
- Personal Claude Code configuration, shared across every machine and every
  per-repo devcontainer.
- A Docker-based devcontainer sandbox for running Claude Code confined to a
  single repo's workspace.

Development context:

- **Personal project**, single author (commits under the names `statox` /
  `Adrien Fabre` / `afabre`, all the same person across several email
  addresses and machines).
- **Mature and continuously active**: first commit in 2014, 998 commits total.
  Recent activity (last few days) concentrates on the devcontainer and MCP
  setup - this is the area currently being iterated on.
- No teams, no external contributors, no issue tracker - all context lives in
  commit history and the [`superpowers/`](superpowers/) planning docs.

#### Modules interactions

Two mostly-independent flows exist side by side:

1. **Dotfile installation.** [`scripts/set-dotfiles.sh`](scripts/set-dotfiles.sh)
   reads [`scripts/files_list`](scripts/files_list) and, for each entry, backs
   up whatever exists at `$HOME/.<entry>` into a fresh timestamped directory
   under [`saved-dotfiles/`](saved-dotfiles/), then symlinks the repo's copy in
   its place. This is how `claude/`, `config/*`, `bin/*`, `bash_aliases`,
   `zshrc`, and `Xmodmap` all end up live on the host.
   [`githooks/setup.sh`](githooks/setup.sh) separately installs
   [`githooks/files/post-merge`](githooks/files/post-merge), which re-syncs
   nvim plugins after a `git pull` changes `lazy-lock.json`.

2. **Devcontainer sandbox.** The `claude` alias (`bash_aliases:394`) runs
   [`devcontainer/scripts/claude-devcontainer`](devcontainer/scripts/claude-devcontainer),
   which brings up the singleton `broker` and `mcp-everything` containers
   ([`devcontainer/docker-compose.yml`](devcontainer/docker-compose.yml)),
   then creates or reuses a per-repo `agent` container via the
   `@devcontainers/cli`
   ([`devcontainer/devcontainer.json`](devcontainer/devcontainer.json)).
   [`devcontainer/agent/postCreate.sh`](devcontainer/agent/postCreate.sh)
   symlinks `claude/*` into `~/.claude` inside the container and jq-merges
   [`claude/mcp-servers.json`](claude/mcp-servers.json) into `~/.claude.json`.

These two flows share one thing: the devcontainer's `agent` container mounts
this whole repo at `~/.dotfiles`, so the files the dotfile-installer flow
manages on the host are exactly what gets symlinked into `~/.claude` inside
the container too.

#### Environment integration

This is a personal setup repo, not a library, backend, or frontend. It
integrates with:

- The local Linux desktop session (i3 + polybar + rofi + dunst + redshift +
  espanso), driven by [`config/i3/config`](config/i3/config) as the hub that
  launches the others.
- The user's shell (zsh with `oh-my-zsh`, `bash_aliases`).
- Docker / Docker Compose and the `@devcontainers/cli`, for the sandboxed
  Claude Code setup.
- Claude Code itself, via `claude/` and the plugins/MCP servers it configures.

There is no external consumer beyond the repo owner's own machines.

### Data Model

This repo has no persisted application data model in the traditional sense.
What it manages is configuration and state files:

- The dotfiles and configs themselves (tracked in git, symlinked to `$HOME`).
- [`saved-dotfiles/`](saved-dotfiles/) - timestamped, gitignored backups of
  whatever previously lived at each managed path, created on every
  `set-dotfiles.sh` run. Purely local, never committed.
- Docker named volumes used by the devcontainer setup (`claude-home`,
  `glab-config`, `vscode-server`, `broker-sock`) hold container-side state
  (Claude session data, `gh`/`glab` auth, VS Code Server) but are opaque to
  this repo - defined only as mount targets in
  [`devcontainer/devcontainer.json`](devcontainer/devcontainer.json).

Nothing here uses a database, ORM, or schema file.

### Patterns

- **Single source of truth for symlinking.** All host dotfiles funnel through
  one manifest, [`scripts/files_list`](scripts/files_list), consumed by one
  script.
- **Lua as the config language for editor/terminal.** Both
  [`config/nvim/`](config/nvim/) and [`config/wezterm/`](config/wezterm/) are
  fully Lua-configured; other tools (`dunst`, `espanso`, `systemd`, `polybar`,
  `redshift`, `btop`) use their own native declarative or shell formats - no
  mixing within a tool.
- **Docker Compose for shared singletons, plain devcontainer CLI for
  per-repo instances.** `broker` and `mcp-everything` are global,
  `restart: unless-stopped` services; `agent` is created/destroyed per
  workspace via `devcontainers up`/`exec`, not Compose.
- **Design docs before big changes.** The devcontainer and MCP-broker work
  went through explicit design -> plan -> status documents in
  [`superpowers/specs/`](superpowers/specs/) and
  [`superpowers/plans/`](superpowers/plans/) before implementation, using
  Claude Code's own "superpowers" skill workflow.
- **Defensive bash.** Scripts and hooks use `set -e`/`set -euo pipefail` and
  guard checks (e.g. [`claude/bell-notify.sh`](claude/bell-notify.sh) no-ops
  if the broker socket is missing) rather than assuming success.
- **Inconsistency: nvim plugin declarations are split without a clear rule.**
  Most plugins live inline in
  [`config/nvim/lua/plugins.lua`](config/nvim/lua/plugins.lua) (365 lines),
  but ten have been extracted into their own files under
  [`config/nvim/lua/plugins/`](config/nvim/lua/plugins/) - there's no
  documented criterion for which plugins get promoted.
- **Leftover migration residue.** A legacy
  [`config/nvim/after/ftplugin/typescript.vim`](config/nvim/after/ftplugin/typescript.vim)
  coexists with its `.lua` replacement - see Tech Debt.

### Area Deep Dives

#### devcontainer/

| Area  |  devcontainer/ |
|-------|------------|
| Scope | Docker-based sandbox for running Claude Code confined to one repo's workspace, plus shared broker/MCP infrastructure. Does not touch desktop configs or the dotfile-symlink flow. |
| Size / complexity | Small (13 files, ~96 KB) but conceptually dense - three containers with different lifecycles. |
| Quality | High - thorough comments explaining *why*, defensive idempotency, documented edge cases. |
| Activity | Very high. `git log --oneline -20 -- devcontainer` covers essentially this directory's entire history; last commit 14 hours ago. This is the most actively-developed area in the repo right now. |
| Concerns | One dangling, commented-out `--build-no-cache` flag in the rebuild script; otherwise clean. |

Two containers with different lifecycles run side by side, wired together over
a Docker network and shared volumes:

```
host
├── broker (global singleton, network_mode: host)      - forwards notifications
├── mcp-everything (global singleton, on mcp-net)       - test MCP server
└── agent (one per repo, on mcp-net)                    - runs Claude Code
```

The `claude` alias (`bash_aliases:394`) drives the whole flow: it re-sources
`nvm.sh` (the alias shadows zsh's lazy-loaded nvm stub), exports `DEV_UID`,
bootstraps `~/.claude.json` and a per-workspace `.venv-claude/`, brings up the
Compose stack, then uses `@devcontainers/cli` to create/reuse the per-cwd
`agent` container. `code-devcontainer` does the same setup, then attaches VS
Code via an undocumented `vscode-remote://attached-container+<hex>` URI - a
documented fallback exists (manual "Attach to Running Container") in case that
breaks.
[`devcontainer/agent/postCreate.sh`](devcontainer/agent/postCreate.sh) runs
inside the agent container: it symlinks `~/.dotfiles/claude/*` into the
persistent `~/.claude` volume, jq-merges `mcp-servers.json` into
`~/.claude.json`, and installs the `typescript-language-server` and configured
plugins.

The [`devcontainer/README.md`](devcontainer/README.md) already contains a
detailed architecture diagram - not duplicated here.

(`claudecode.nvim`, `codecompanion.nvim`):

```
nvim/

  lua/
    options.lua, mappings.lua, autocommands.lua, commands.lua,
    motion.lua, statusline.lua, abbreviations.lua, diffsettings.lua,
    colorscheme_custom.lua, colorscheme_watcher.lua
```

`notes/` contains self-written migration docs (autoload-vimscript -> Lua,
packer -> lazy.nvim), evidence of deliberate, documented refactors rather than
ad hoc changes.

#### Dotfile-management tooling (scripts/, githooks/, bin/, saved-dotfiles/)

| Area  |  Dotfile-management tooling |
|-------|------------|
| Scope | Installing/backing-up dotfiles (`scripts/`), the post-merge git hook (`githooks/`), and a grab-bag of standalone desktop utility scripts (`bin/`). `saved-dotfiles/` is generated output, not managed logic. |
| Size / complexity | Small (~350 lines across `bin/`; `set-dotfiles.sh` ~70 lines; `post-merge` ~20 lines). Low complexity, straightforward bash. |
| Quality | Reasonable - has comments, `set -e`, basic guards - but no tests, no backup pruning, and some duplication (e.g. `monitor-battery`'s battery-path lookup repeated across helper functions). |
| Activity | `scripts/` last touched 3 months ago; `bin/` and `githooks/` are more recently touched (battery tooling, a systemd fix). |
| Concerns | `saved-dotfiles/` grows unboundedly - see Tech Debt. |

[`scripts/set-dotfiles.sh`](scripts/set-dotfiles.sh) runs unconditionally on
every invocation: for each entry in
[`scripts/files_list`](scripts/files_list) that exists under `$HOME`, it
copies it into a fresh `saved-dotfiles/<timestamp>/` directory, deletes the
original, then symlinks the repo's copy in its place - no diff/change
detection, no pruning of old backups.
[`githooks/files/post-merge`](githooks/files/post-merge) diffs
`ORIG_HEAD..HEAD` and, if `config/nvim/lazy-lock.json` changed, runs
`nvim --headless "+Lazy! restore" +qa`.

`bin/` scripts, one line each:

- [`changeVolume`](bin/changeVolume) - `pactl` wrapper for volume up/down/mute, capped at 120%.
- [`dmenu-i3-window-jumper.sh`](bin/dmenu-i3-window-jumper.sh) - dmenu-based i3 window switcher (vendored from an external repo).
- [`i3-lock`](bin/i3-lock) - locks the screen via `i3lock-fancy` + a blurred `scrot` screenshot.
- [`i3-lock-suspend`](bin/i3-lock-suspend) - locks, then suspends after a delay.
- [`mm`](bin/mm) - tiny mouse-jiggler loop to keep the session awake.
- [`monitor-battery`](bin/monitor-battery) - notifies on low/discharging battery, run via the `monitor_battery` systemd timer.
- [`rofi-calc`](bin/rofi-calc) - rofi/dmenu calculator front-end with clipboard copy.
- [`timer`](bin/timer) - pure-bash stopwatch/countdown.
- [`toggle_theme`](bin/toggle_theme) - flips light/dark theme across wezterm, nvim, GTK, and gsettings.

### Setup

Documented in [`README.md`](README.md):

1. Clone to `~/.dotfiles`.
2. Run [`githooks/setup.sh`](githooks/setup.sh) to install the post-merge hook.
3. Edit [`scripts/files_list`](scripts/files_list) to choose which files/dirs
   to manage.
4. Run [`scripts/set-dotfiles.sh`](scripts/set-dotfiles.sh) to back up
   existing dotfiles and symlink the repo's copies into `$HOME`.

For the devcontainer sandbox, setup is: have Docker and `bash_aliases` sourced,
then run `claude` (or `code-devcontainer`) from inside any target repo. Deeper
setup detail (volumes, networks, images) is documented in
[`devcontainer/README.md`](devcontainer/README.md) - not repeated here.

There are no test or lint setup steps - none exist in this repo.

### Build Process

There's no application build in the traditional sense. Two things resemble a
build step:

- **nvim plugins**: `lazy.nvim` installs/updates plugins according to
  [`config/nvim/lua/plugins.lua`](config/nvim/lua/plugins.lua) and
  [`config/nvim/lazy-lock.json`](config/nvim/lazy-lock.json); the post-merge
  hook re-syncs this automatically after a `git pull` changes the lockfile.
- **Devcontainer images**: `docker compose build` / `devcontainers up` build
  the `agent`, `broker`, and `mcp-everything` images from their respective
  Dockerfiles under [`devcontainer/`](devcontainer/), producing containers
  used interactively on the host - not deployed or published anywhere.

### Tests

There is no test suite anywhere in this repo, and none is expected for a
personal dotfiles setup. Correctness is verified manually (per the repo
owner's own `CLAUDE.md` workflow rules, which call for explicit verification
before claiming completion).

## Repo High Level Analysis

### Quality & Improvement Areas

- **`devcontainer/` and `claude/` are the strongest areas**: clean, well-
  commented, and backed by explicit design docs
  ([`superpowers/specs/`](superpowers/specs/)) before implementation. Both are
  actively maintained and show no significant issues.
- **The dotfile-symlink flow is simple and appropriately scoped** for a
  personal repo - one manifest, one script - but has a recurring weak spot:
  `saved-dotfiles/` backups are created unconditionally on every run with no
  pruning, and git history shows past bug-fix commits specifically in this
  logic (e.g. a fix for saving nested items). This is the one area that would
  benefit from tightening.
- **nvim config is functional but shows migration residue**: plugin
  declarations are split between one monolithic file and several extracted
