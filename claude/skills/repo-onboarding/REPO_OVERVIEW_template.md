## Repo High Level Analysis

### Quality & Improvement Areas

An assessment of shape, not a bug hunt. Comment on both what's weak and what
works well:

- Simplicity/readability of the codebase overall.
- Whether separation of concerns is clean or areas are tangled.
- Whether similar problems are solved consistently across the codebase or
  reinvented per area - a strong signal from the Area Deep Dives' cross-area
  comparison.
- How well architecture/module boundaries match what the code actually does.
- Anything about setup, tests, or docs from earlier sections that would trip
  up a newcomer.
- What's notably stable, well-tested, or well-designed - not everything
  flagged here needs to be a problem.

Where a problem is flagged, say briefly why it matters and, if obvious, what
direction a fix would take - don't spec out the fix in detail, that's a
separate task.

### Ongoing Work

Describe any large changes currently in progress in the repo - a migration, a
code extraction, a new feature being built out, a framework upgrade. Signals
to look for:

- Two implementations of the same thing coexisting (old and new auth
  systems, old and new API versions).
- TODO/FIXME comments that reference a ticket, a migration, or "remove after
  X".
- Recent commits or branches whose messages describe a multi-step effort
  still underway.
- Partially wired-up code: a new module referenced in only some places, a
  feature flag gating incomplete work.

If nothing is currently in progress, say so rather than inventing something.

### Tech Debt

Describe known pain points and leftovers that haven't been addressed yet -
not a fresh audit, but what the repo itself already flags or what surfaced
repeatedly during exploration:

- TODO/FIXME/HACK comments describing a known shortcut or missing piece.
- Deprecated code paths, dead code, or disabled/skipped tests still present.
- Workarounds mentioned in code comments, commit messages, or docs, and why
  they exist.
- Known limitations documented in issues, ARCHITECTURE.md, or similar, that
  are still unresolved.

Where possible, note whether the codebase treats each item as acknowledged
(tracked in an issue, commented with intent) or just silently accumulating.

### Change Frequency & Hotspots

Describe which parts of the codebase change most often and where bugs recur -
a signal of recurring pain points for maintainers, not by itself a verdict on
code quality:

- List the files/directories that changed most often over the last year, and
  note if they cluster in one or two Area Deep Dives areas.
- Note who is most active in each hotspot area - the goal is to understand if
  ownership and knowledge is shared by several people or mostly single contributors.
- List where bug-fix commits concentrate (files/directories most often
  touched by commits mentioning a fix or bug).
- Flag any file or area that shows up in both the change-frequency list and
  the bug-fix list - that overlap is the strongest signal of a recurring
  pain point, stronger than either list alone.

If nothing stands out (activity spread evenly, no clustering, no bug
concentration), say so rather than forcing a finding.
